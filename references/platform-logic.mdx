## MVP Platform Thesis

The MVP should make Analog easy to understand at a glance:

```text
Sponsor budget -> controlled spend -> host supply -> proof -> two-way review -> next allocation
```

The active `cc-analog` checkout currently runs the first Host/supply side of that loop:

```text
public directory
-> listing claim or creation
-> Host dashboard
-> discovery, volunteer, funding-readiness, and grant workflows
-> sponsor-ready proof fields
```

The preserved full-platform docs and archived code carry the next Sponsor/spend side of the loop:

```text
Sponsor campaign
-> application portal
-> Host acceptance
-> collab
-> spend card
-> proof, review, and renewal
```

Keep these stages distinct. Current how-to pages can use the active directory/Host routes directly. Sponsor campaign, Stripe Issuing, Sponsor AI, Gmail-style notifications, and full dashboard routes are staged product logic until restored into the active workspace.

The Sponsor experience should feel agentic even when the user is not literally chatting with an agent. Every Sponsor session should try to understand why the Sponsor is here, what job they are trying to finish, and which UI, settings, or recommendations should appear next.

The platform is testing which product conventions make eComm brands, SaaS/tech companies, civic/public/nonprofit funders, political action Sponsors, and partner businesses comfortable giving Analog budget to deploy on their behalf. The work starts with spend infrastructure, but the goal is marketplace enablement: enough structured Sponsor demand and Host-side supply for Analog to observe what repeats.

The platform should prioritize cash-based Host enablement over product-only gifting. Product can be a campaign asset, but the core learning surface is how cash, services, tools, proof, and renewal help Hosts create better gatherings.

The platform should also capture partnership quality. Sponsors and Hosts should review each other at closeout and every 90 days for long-running collabs, so future approvals reflect responsiveness, spend stewardship, clarity, respect, admin burden, and whether the relationship should repeat. This is how Analog keeps good actors on-platform, constrains or removes bad actors, and reinforces trusted behavior through proof. Prior checks written by Sponsors and checks received by Hosts should appear as public-safe funding history when permissioned.

The Host-side platform posture is "go pro," not "make an invite." A community builder should see Analog as the place where their work becomes visible, fundable, and more professional: local identity, profile, proof, media inclusion, funding opportunities, and next-dollar logic.

For high-ticket sponsorships, the platform direction expands from card spend to sponsorship governance. Analog should still use spend cards for distributed Host and vendor budgets, while using files/docs, contracts, invoices, milestones, risk registers, proof gates, and AI-assisted recommendations to keep large IRL investments safe.

For vendor and tax records, the platform direction is simplification: Hosts, Venues, contractors, photographers, caterers, artists, local operators, and service providers should not create a new back-office mess for Sponsors. Analog should keep reusable vendor/payee records, tax-document status, duplicate detection, and 1099 support context where needed, with production filing or withholding handled only through approved tax/payment rails.

For public and nonprofit funding, the platform direction is downstream fund administration: eligibility, restricted uses, conflict checks, public-safe proof, fraud/waste/abuse review, support needs, and funder-ready reports around the same campaign, event-set, collab, spend, files, and proof objects. The Host-facing grant experience should be guided: one reusable application, local grant matches, short grant-specific follow-up flows, background packet/form preparation, and explicit Host approval before submission.

For political action funding, the platform direction is compliance-safe field fund administration: authorized entities, national/state/district/county/city/ward/event sets, approved uses, privacy boundaries, field proof, sensitive-affiliation controls, and allocation rollups around the same core objects. Treat this as a high-control use case, not generic event marketing.

## IRL Culture Data Opportunity

Analog should compound a data set that existing digital and B2B systems do not own well.

<Info>
Use ZoomInfo and Meta as strategic analogies, not literal product comparisons. ZoomInfo-style data makes companies, contacts, and accounts actionable. Meta-style social data makes digital creators and audiences actionable. Analog's opportunity is to make the fluid motion of IRL events, culture, and community actionable for marketing, audience allocation, and sponsor-funded market growth.
</Info>

The platform should treat every campaign, application, event set, hosted page, proof packet, two-way review, city lead note, owned IRL room, and sponsor renewal decision as product memory.

| Data layer | What Analog can learn | Product expression |
|---|---|---|
| Event and ritual motion | Which gatherings recur, grow, fade, travel across cities, or create category momentum. | Host events, hosted pages, catalog records, proof summaries. |
| Cultural operator graph | Which hosts, spaces, creators, restaurants, dot connectors, and city leads can move a room. | Host profiles, connector profiles, applications, city lead referrals. |
| Audience potential | Which communities resemble a sponsor's best customers, strategic accounts, civic audience, or category-growth opportunity. | Audience graph, event-set fit, lookalike event recommendations, sponsor data summaries. |
| Mention performance | Which Hosts and Venues create visible IRL signal relative to attendance, spend, and Sponsor fit. | Authorized Host mentions, Venue mentions, ratios, percentiles, proof summaries. |
| Partnership reliability | Which Hosts and Sponsors are clear, responsive, respectful, low-friction, and worth repeating together. | Two-way reviews, closeout notes, 90-day checkpoints, renewal decisions. |
| Sponsor fit and renewal | Which sponsor perspectives, budget shapes, spend rules, and proof standards earn the next allocation. | Campaigns, event sets, collabs, spend cards, proof, unlock decisions. |

This is not a plan to sell raw attendee data. The defensible asset is privacy-safe, provenance-aware IRL market intelligence: who gathers, why they gather, who can host, what sponsors should fund, what proof is credible, and where the next dollar should go.

## Public Network Posture

Analog should be explicit that the network is designed for opportunity exposure. A Host, Venue, operator, connector, or organizer joins Analog partly so Sponsors, public/nonprofit funders, partner platforms, media, and approved agentic workflows can find them.

Default discoverability should apply to public-safe opportunity data:

- Host, Venue, operator, event, community, and recurring ritual identity.
- Geography, category, cadence, audience hypothesis, and market fit.
- Funding needs, service/tool needs, Sponsor preferences, exclusions, and acceptable brand presence.
- Proof/readiness status, claim status, provenance, and next action.

This is not an open listing promise and not a raw data export. Discovery can happen through Analog search, Sponsor-side workflows, Event set prospecting, partner APIs, and Agent MCP once those rails exist. Any user or sensitive record can opt out or require private, link-only, compliance-sensitive, or human-reviewed visibility.

The product should make the bargain clear in onboarding, claim, application, and settings flows:

```text
Analog uses your public-safe profile and activity data
to help route relevant funding, partnership, sponsorship,
grant, media, and custom workflow opportunities to you.
```

Active approximations are public directory listings, community pages, claim flows, bookmarks, reviews, volunteer roles, grant applications, listing search, and Host opportunity preferences. Preserved full-platform approximations are public hosted event pages, public application portals, public pitch pages, Host opportunities, Sponsor AI assistant context, managed outreach, Event Prospecting Catalog records, files/docs, and feature/settings preferences. Future platform-as-a-service approximations would let event-page, ticketing, social experience, venue, club, creator, or community platforms use Analog's Sponsor funding, proof, attribution, and opportunity-routing layer while keeping their own user experience. Do not imply production public APIs, external Agent MCP access, partner platform dashboards, embedded funding workflows, autonomous outreach, or fully open catalog export until those rails are built.

## Sponsor-Shared Campaign Dynamics

The most important marketplace behavior is a sponsor sharing a campaign, fund, or program as a visible demand signal.

```text
Sponsor shares campaign -> market sees money on the table -> hosts respond -> Analog governs spend -> proof selects winners
```

This does not mean making every sponsor budget public or promising funding to anyone who sees the link. It means turning hidden sponsor interest into a clear, shareable ask with eligibility, spend rules, proof standards, and a path to apply or be invited.

<Steps>
  <Step title="Declare demand">
    The sponsor frames the campaign as a concrete market ask: fund this kind of moment, for this audience, in this place, under these rules.
  </Step>

  <Step title="Let the ask travel">
    The campaign link, brief, or fund page moves through sponsor channels, city leads, partners, `The Connector`, social, and owned Analog gatherings.
  </Step>

  <Step title="Create supply response">
    Hosts, spaces, connectors, restaurants, operators, and community builders apply, refer, or package activation ideas because the opportunity is specific enough to act on.
  </Step>

  <Step title="Govern the money">
    Analog approves the right operators, sets allocations and spend limits, captures proof, and protects sponsor trust.
  </Step>

  <Step title="Select the next dollar">
    Proof and two-way reviews decide which hosts earn renewal, which cities deserve expansion, which spend rules should change, and which campaign shapes should stop.
  </Step>
</Steps>

| Dynamic | Product surface | Marketplace effect |
|---|---|---|
| Visible budget signal <Tooltip headline="Visible budget signal" tip="A sponsor-backed campaign, fund, or program that makes the available opportunity legible without exposing unrestricted money or private budget details."><span className="analog-help">?</span></Tooltip> | Campaign, event set, application portal, hosted form, campaign copy | Hosts understand that real sponsor capital exists for a specific mission, city, audience, or format. |
| Shareable demand beacon <Tooltip headline="Demand beacon" tip="A campaign or fund page that can be shared by sponsors, city leads, partners, media, and owned IRL attendees to invite qualified supply."><span className="analog-help">?</span></Tooltip> | Application link, sponsor outreach, files/docs, hosted page, partner surface | The market can route the right hosts, spaces, connectors, and communities toward the opportunity. |
| Supply response | Applications, invited hosts, prospecting replies, city lead referrals, pitch docs | Analog sees what the market produces when money and rules are visible. |
| Spend-backed selection | Collabs, allocations, spend cards, categories, transactions, proof | The best operators are selected by execution, spend stewardship, and sponsor-believable proof, not only relationship access. |
| Repeat campaign sharing | Renewal packet, unlock recommendation, sponsor data, proof recap | The sponsor can share a sharper next campaign because the platform learned who responded and what worked. |

The "windfall" is the sponsor budget that is already looking for trusted IRL channels but cannot move because the supply, governance, and proof layer is missing. Analog's job is to make that capital actionable without making it careless.

## Campaign Visibility Modes

A campaign should choose its marketplace visibility before it starts routing supply.

Use [Marketplace Logic](/platform/marketplace) for the full doctrine.

| Visibility mode | Use when | Active or staged approximation |
|---|---|---|
| Private link-only | The Sponsor has known Hosts, a sensitive brief, a private audience, or an early test. | Application portal link, direct invite, files/docs, warm outreach, invited applications. |
| Broadly marketed | The Sponsor wants a public fund, grant, raffle, or city/category signal that pulls supply forward. | Campaign copy, public application portal, media CTA, regional dashboard placement, newsletter/social/owned IRL distribution. |
| ICP-targeted | The Sponsor wants the ideal Host, Venue, community, or audience profile rather than the whole market. | Event set prospecting, managed outreach, Event Prospecting Catalog, city lead lists, partner lists, lookalike event recommendations. |

The product should make the visibility choice practical:

```text
Who can see this campaign?
Who should be invited?
What kind of supply is ideal?
What proof benchmark will decide renewal?
```

## Hosted Event Page Posture

Hosted event pages are useful, but they are not the company thesis by themselves.

The market is filling with carbon-copy event pages: good-looking invitations, RSVP flows, ticket widgets, calendars, reminders, and lightweight discovery. That front door will keep getting cheaper. Analog should not treat a generic event page as the value unlock.

The value unlock is turning the event page into a community brand asset.

| Commodity event page | Analog event asset |
|---|---|
| Title, date, image, RSVP, ticket link. | Host story, community identity, Venue context, Sponsor role, attendee benefit, proof path, QR/claim flow, and event history. |
| Helps people find or join one event. | Makes the gathering feel worth attending and makes the Host more fundable afterward. |
| Looks similar across many events. | Can feel like a small Coachella or Outside Lands-style web experience for the community when the moment deserves it. |
| Creates shallow traffic and logistics data. | Creates permissioned RSVPs, claims, attendance signal, mention context, recap material, and proof for renewal. |
| Can support generic ads. | Can support better future ad constructs because the page has real story, fit, consent, and proof attached. |

The platform rule:

```text
Do not compete on prettier event pages alone.
Use event pages to convert attendance,
make the Host and community look fundable,
and turn the gathering into proof.
```

The page should feed the Host's professional record. A strong page should make the Host easier to include in a city guide, campaign material, Sponsor package, grant packet, directory, or renewal memo after the gathering is over.

Active surfaces approximate this through public community pages, listing events, reviews, bookmarks, claim links, media, grant-readiness fields, and directory search. Preserved full-platform surfaces approximate it through hosted pages, themes, backgrounds, design kits, custom CSS, custom HTML, RSVP fields, ticket types, QR or claim links, files/docs, and proof packets. Future advertising or retargeting work must stay permissioned and privacy-safe; do not imply raw attendee data, pixels, or paid-media automation exists unless it is actually configured.

## Mention Coverage Upsell

When a Sponsor saves a campaign that depends on a Venue, the platform should prompt an optional attribution-coverage upsell:

```text
Increase attribution coverage by adding a Venue reward.
```

The Sponsor can set a flat reward, such as a planning default of USD 200, for a Venue that authorizes campaign-window social listening or connects eligible social accounts.

| Product rule | Why it matters |
|---|---|
| Keep the reward flat and explicit. | The Venue knows what it earns for authorization. |
| Keep it separate from Host allocation. | Host upside is not reduced to buy Venue coverage. |
| Prompt at campaign save. | Sponsors decide proof coverage before the activation starts. |
| Give the Host a clean script. | The Host can offer the Venue a benefit instead of asking for unpaid monitoring help. |
| Treat mentions as proof assets. | Host mentions, Venue mentions, attendance, and Sponsor mentions become benchmarkable. |

This is future product direction until automated Venue authorization, payout, tax, and platform-connection rails exist. Active work can approximate the public record side with listing locations, media, reviews, and Host-owned listing content. Preserved full-platform work can approximate the campaign side with campaign settings, budget notes, files/docs, Host event location, campaign content, story mentions, tasks, and Sponsor data.

## Owned IRL As Market Seeding

Analog-owned gatherings should sit beside sponsor-shared campaigns as the company's own supply-creation engine.

The operating rule is simple: run owned IRL in the same city, category, or sponsor-perspective lane as the campaign. The room gives city leads a trusted way to explain the ask, hear blockers, collect referrals, and route interested people into product objects.

| Owned IRL motion | How it talks to shared campaigns |
|---|---|
| Community-builder happy hours | Recruit hosts, connectors, and spaces into live campaign/fund opportunities. |
| Sponsor-listening salons | Turn sponsor demand into clearer campaign briefs, proof rules, and budget conventions. |
| City lead dinners or walks | Surface trusted operators and let them hear the money-on-the-table ask directly. |
| Connector interviews and live recordings | Convert cultural visibility into sponsor-safe profiles, applications, and Activation Opportunities. |
| Fund launch gatherings | Make a sponsor-backed program feel real in-market, then route attendees and referrals into the application flow. |

Owned IRL should never be separate from the product. Every gathering should point to a campaign, fund, host profile, application portal, catalog record, proof asset, sponsor lead, or asset-need note.

## Sponsor UX Rules

These rules apply to the preserved full-platform Sponsor workspace. The active `cc-analog` checkout does not currently expose live Sponsor campaign dashboards, Sponsor AI, collab management, or spend-card operations outside archived code.

Sponsor screens should stay sponsor-centric:

- Sponsor sessions should feel like an agentic workflow whether the Sponsor is in chat, a form, a dashboard, a data view, or settings.
- The first job of each Sponsor session is to understand intent: create a campaign, test market demand, find supply, review applicants, inspect proof, adjust budget, route vendor/admin work, or decide renewal.
- The UI should offer simple clicks and inputs first, then render the next useful action, explanation, field, setting, or recommendation.
- Advanced features can stay hidden until they make sense. Reveal Venue rewards, visibility modes, Agent MCP/API sourcing, vendor records, high-ticket controls, public/nonprofit controls, political controls, or proof benchmarks when the Sponsor's context asks for them.
- Deeper digging should not hit a dead end. If the Sponsor keeps asking "why," "who else," "what if," or "show me more," the product should route to more data, settings, recommended workflows, files/docs, or a human approval step.
- Agentic UI must still respect approval boundaries. AI can recommend, summarize, draft, rank, and prepare; humans approve publishing, outreach, private data sharing, spend, contracts, sensitive claims, and funding decisions.
- Campaigns are the main paid/funded object and the first spend-infra learning container.
- Campaign settings should capture mission, market persona, main value-add, start date, special requests, host visibility, mailing-address requirement, and auto-renew preference.
- Campaign start dates should be visible to applicants so future campaigns set clear expectations about when review, funding, or activation starts.
- Sponsors should be able to post future-dated campaigns and use the first 48 hours of application response as a demand signal before scaling operator work.
- Campaign save should prompt for a Venue reward when Venue mention coverage would improve attribution.
- Campaigns and collabs should prompt a two-way review when funds are spent, the campaign ends, or the collab closes; long-running collabs should prompt a lightweight review every 90 days.
- Event sets split a campaign by market/persona, special requests, files/assets, and prospecting lane. Treat them as demand lanes that can pull supply into the market.
- Application portal collects standardized host pitches and keeps attachments/answers structured.
- Sponsor applications should avoid paywalling before value is visible: show the first five applicants clearly on free, blur or limit the rest, and use first accepted applicant as the conversion moment into a Sponsor plan with credits, campaign commitment, or program fee.
- Sponsor outreach routes external leads back to the same application workflow, so managed GTM creates product-readable supply.
- Campaign detail shows campaign metrics, event sets, files, sponsor-visible proof, sponsored hosts, and UGC.
- Sponsor-host detail owns host-level campaign activity, shared files, asset requests, and shipment logs.
- UGC and assisted outcome summaries belong on sponsor/data surfaces.

## Host UX Rules

Host screens should stay simple, snappy, and funding/proof-first:

- In the active Host app, the dashboard sections are "What's new," "Get discovered," "Get volunteers," and "Get funded."
- Active Host workflows focus on claiming or creating listings, improving the public listing record, managing volunteer interest, applying to the Analog founding grant, and preparing for sponsorship.
- In the preserved full platform, events are the home/default view.
- Do not make Hosts feel like they are operating a deep admin cockpit. Show the next useful action, then tuck advanced settings away until needed.
- The Host home should become region-aware. If a Host belongs to New York, the experience should be able to show `AnalogNYC` and prioritize local funds, perks, media requests, and proof tasks.
- The Host home should eventually include the universal Analog application, a geo-dashboard grant shelf, and guided grant-specific follow-up flows so Hosts can reuse one record across public grants, private grants, Sponsor funds, and community collective funds.
- Hosts can create events before sponsorship lands.
- Host event creation can attach one sponsor budget at creation, or no sponsor yet.
- Host event setup should show whether the Sponsor added a Venue reward or mention-coverage request.
- Hosted event pages should not default to a generic calendar card for fundable moments; they should help the Host express the community, convert attendance, capture permissioned proof, and preserve event history.
- Sponsor budgets and sponsor cards live in the host Sponsors view.
- Spend cards and transactions live on event and sponsor budget detail surfaces.
- Hosts can request mailed assets from event set assets.
- Hosts should see enough budget, task, file, and spend detail to execute.
- Hosts should get a lightweight review prompt at closeout and every 90 days for long-running collabs, focused on Sponsor clarity, respect, responsiveness, admin burden, and future fit.
- Hosts should not see sponsor analytics-heavy UGC grids.

## Application Portal Rules

The public application portal should:

- show running campaigns
- show campaign start date and expected review or funding window
- use structured pitch fields
- collect community/host identity
- collect pitch title, description, requested amount, socials, expected attendees, and attachment metadata
- support future ranking by keeping fields structured

File attachments are metadata only today.

## Files, Docs, And Messages

Files are now part of the operating product, not just storage:

- Sponsor and host workspaces split files into Analog Docs, uploaded files, and shared files.
- Analog Docs can be manual, template-generated, or deep-research generated.
- Files can be assigned to sponsors, hosts, campaigns, event sets, host events, collabs, applications, or external recipients.
- DM threads are available for active partnerships and pitch contexts.
- Pitch sharing can create a host-owned Analog Doc and make it visible to a sponsor or external recipient.

## Outreach And Prospecting Rules

Keep two related concepts distinct:

- Sponsor Outreach is the sponsor workspace page for managed email/direct mail outbound and future self-serve controls.
- Event Set Prospecting is a campaign detail action that starts external prospecting for one event set.

Both should route interested hosts back to the campaign application link so leads enter the same approval workflow.

Managed outreach, city lead sourcing, media, and brand-fund GTM are allowed in the MVP because they create supply the marketplace can observe. They should always produce product objects: applications, event sets, host profiles, pitch docs, proof, asset requests, or no-go learning.

City-level supply work should be tagged by market whenever possible. The goal is not only to close one campaign; it is to learn whether AnalogNYC, AnalogSF, or a future city layer is becoming a repeatable operating department.
