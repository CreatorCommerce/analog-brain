| Layer | Source in `cc-analog` | How docs should treat it |

The active product spine is:

```text
public directory record
-> claim or create listing
-> Host organization owns listing
-> Host improves discovery, volunteers, funding readiness, and grant applications
-> product learns which communities, categories, and proof fields are sponsor-ready
```

The preserved full-platform spine is:

```text
Sponsor demand
-> campaign or application portal
-> Host application
-> invited or active collab
-> governed spend through a spend card
-> proof, review, and renewal
```

When a page says "current," check which layer it means. Do not describe archived sponsor dashboards, live Stripe Issuing flows, Sponsor AI, Gmail OAuth, public Agent MCP, or full marketplace APIs as active until they are restored into the active workspace.

## MVP Boundary

Analog's full platform logic is an MVP for Sponsors and Hosts to fund, operate, and prove community-led IRL activations through structured campaigns, host events, application portals, files/docs, managed outreach, hosted event pages, and temporary operational spend cards.

The active `cc-analog` checkout is narrower today: it runs the public Analog directory, the Host claim/create flow, Host dashboard surfaces, organization-owned listings, public reviews, bookmarks, volunteer recruitment, founding-grant applications, natural-language directory search, media/storage, notifications, and Instagram connection intent. The sponsor-host-card platform is preserved in the archive and should be treated as staged product logic.

The wedge is governed IRL spend: Sponsor budget becomes campaign structure, Host-side execution, shared spend, proof, reporting, sentiment, and renewal logic. The product is still an MVP and is not yet running production-grade billing, payout, payment, or full sponsor-side infrastructure. Active Host auth exists through WorkOS AuthKit when configured.

The platform focus is fintech enablement plus marketplace enablement. Analog is not trying to become a bank in the MVP. It is shoring up the spend infrastructure, campaign conventions, Host approval flow, proof loop, and supply-creation motion needed to earn the right to deploy community activation investments on behalf of brands and businesses.

The strategic funding model is cash-based Host enablement. Product, samples, and merch can support a collab, but the core product should learn how cash, controlled spend, services, tools, proof, and renewal help Hosts accomplish their actual community goals.

The strategic proof model is mention-aware. Public Host and Venue mentions should become proof assets when they are authorized and campaign-relevant. Analog should help Sponsors understand which rooms, spaces, Hosts, and categories created visible IRL signal instead of letting mentions disappear into recap screenshots.

The strategic review model is two-sided. Sponsors should review Hosts, and Hosts should review Sponsors, at closeout and at 90-day checkpoints for long-running collabs. Human review fills the gap between AI proof, spend records, and social listening: whether the partner was responsive, respectful, clear, trustworthy, and worth repeating. The product job is to keep good actors on-platform, constrain or remove bad actors, and reinforce strong behavior through proof. Prior checks written by Sponsors and checks received by Hosts are part of that trust signal when shown as permissioned funding history.

The hosted event page posture is not "win because we have event pages." Basic event pages are becoming commodity front doors. Analog should use hosted pages as community brand assets: high-context surfaces that make the gathering feel worth attending, express the Host and Venue, explain the Sponsor's additive role, capture permissioned RSVPs or claims, and preserve proof for renewal.

The aspirational event-page standard is closer to a small Coachella or Outside Lands-style web experience for a trusted community than a carbon-copy calendar card. The preserved full-platform docs include primitives for this through hosted event pages, themes, backgrounds, RSVP fields, ticket types, custom CSS, custom HTML, QR assets, and proof capture. Active `cc-analog` currently emphasizes directory community pages, listing events, reviews, bookmarks, grant readiness, and claim flows. Future ad and retargeting constructs should build from permissioned event or community assets, not from raw attendee data or generic invite traffic.

The Host-side product direction is professionalization, not event execution alone. A Host's Analog surface should become a professional record: community story, event history, proof, media inclusion, funding asks, Sponsor fit, public-safe discoverability, and unlock history. Event tools help the Host execute one gathering; Analog should help the Host build a career path around gathering people.

Public and nonprofit programs are a high-stakes version of the same model: mission capital stays accountable upstream while governed budgets, services, and proof move downstream to Hosts, community groups, small businesses, spaces, and local operators.

The NYC pre-seed Sponsor cohort is a current operating motion, not a finished product surface. Analog should track it with LOIs, refundable deposits, spreadsheets, files/docs, Event sets, Sponsor briefs, and proof packets while the product learns the workflow. Do not imply the MVP already has native LOI, deposit, billing, or cohort CRM rails.

Public-space and social connection programs add a measurement layer. When a Sponsor funds parks, plazas, libraries, corridors, cleanups, stewardship, wellness access, arts, youth, sports, or civic programs, Analog should be able to store public-safe proof of belonging, connection, trust, agency, respect, meaningfulness, and cross-group contact without overclaiming long-term outcomes. Use [Public Space Social Impact Measurement](/references/public-space-social-impact-measurement) for the reference pattern.

Public-space activations add a readiness layer before measurement. When a Host wants to activate a park, plaza, sidewalk, street, Open Street, public housing ground, schoolyard, or other public-facing space, Analog should capture permit path, event risk tier, insurance or COI needs, site-plan needs, weather plan, vending/revenue activity, and official source links before funding or renewal. Use [Public Space Activation Infrastructure](/references/public-space-activation-infrastructure) for the reference pattern.

Political action programs are another high-stakes version: campaign or advocacy capital stays accountable upstream while governed budgets, approved uses, privacy boundaries, and compliance-safe proof move downstream to trusted local Hosts. Treat this as a strategic use case layered onto the same MVP objects, not as a claim that Analog currently provides production campaign-finance infrastructure.

## IRL Culture Graph Primitive

Analog's product memory should be more than event records. The larger primitive is an IRL culture graph: a privacy-safe, provenance-aware map of events, recurring rituals, cultural operators, venues, audience hypotheses, sponsor fit, spend behavior, authorized mentions, two-way reviews, proof quality, and renewal logic.

The strategic analogy is simple:

| Data category | What it makes actionable |
|---|---|
| B2B account data | Companies, contacts, roles, and prospecting paths. |
| Digital creator data | Creator reach, digital audiences, content performance, and social distribution. |
| Analog IRL culture data | Which real-world communities gather, who can move them, what sponsors they fit, and which funded moments create audience or category potential. |

In the MVP, this graph is assembled from product objects rather than a standalone data product: campaigns, event sets, applications, host profiles, hosted pages, spend cards, transactions, files/docs, proof packets, review notes, city lead notes, media capture, and renewal decisions.

## AI Memory And Context Primitive

AI memory should help Analog learn without replacing product records.

Use [AI Memory And Context](/platform/ai-memory-and-context) as the plain-English Platform page. The product rule is:

```text
Product data is the source of truth.
AI memory is a small, sourced note about what helps the assistant work.
Context is the compact packet an agent gets for one task.
```

The preserved full-platform docs approximate this through Sponsor AI context packets, files/docs, Analog Docs, deep research docs, city tactics guide categories, campaign summaries, and sponsor-visible workspace data. The active product already has a smaller search-context pattern through directory `SearchConversation`, intent summaries, embeddings, and public-safe listing DTOs. Future product work should add scoped AI memory, recursive knowledge metadata, Market Agent snapshots, source ids, confidence labels, review flows, and version history before treating market or culture learning as durable assistant context.

Growth AI may use internal Analog doctrine, public sources, Market Agent research, public-safe opportunity data, and approved aggregate product trends. It should not use raw private product data by default.

## Agentic Sponsor Experience Primitive

The Sponsor-side product should feel like an agentic workflow even when the interface is mostly buttons, forms, cards, and tables.

The primitive is the session job:

```text
understand why the Sponsor is here
-> show the smallest useful next step
-> reveal deeper controls only when context asks for them
-> keep drilling into data, settings, or recommendations when needed
-> route sensitive actions to human approval
```

This is not only a chat assistant. Chat, dynamic UI, forms, dashboards, settings, application inboxes, and proof views should all act like one guided workflow.

| Session job | Current approximation | Rule |
|---|---|---|
| Understand intent | Preserved full-platform surfaces: onboarding role/preferences, Sponsor AI assistant prompt, campaign dashboard state, selected campaign, application inbox, Sponsor data view | Ask or infer whether the Sponsor is here to create a campaign, find supply, review applicants, inspect proof, adjust budget, fix admin work, or decide renewal. |
| Render the next useful step | Campaign creation fields, event set setup, application cards, proof summaries, files/docs, settings/features | Keep first actions simple. Then show the next field, recommendation, setting, or workflow based on context. |
| Reveal advanced controls progressively | Feature setup, campaign settings, Venue reward notes, visibility mode, vendor/payee context, public/nonprofit controls, political controls, high-ticket files/docs | Hide features until they matter. Do not make every Sponsor configure every possible rail up front. |
| Support deep drill-down | Sponsor data summaries, proof packets, application detail, Event Prospecting Catalog, files/docs, deep research docs, future Market Agent snapshots | If the Sponsor keeps asking why, who else, what if, or show more, keep routing them to deeper evidence, options, or settings. |
| Preserve approval gates | Tasks, files/docs, campaign save, applicant acceptance, collab creation, outreach drafts, spend controls | AI can recommend, draft, rank, summarize, and prepare. Humans approve publishing, outreach, private data sharing, spend, contracts, sensitive claims, and funding decisions. |

The Host-side product should not mirror this depth by default. Hosts need a fast surface for money, tasks, events, proof, opportunities, and profile upkeep. Advanced settings should appear only when they help the Host get funded, execute, or protect sensitive visibility.

## Public Opportunity Graph Primitive

The IRL culture graph should have a public-safe opportunity layer. Analog is intentionally not a closed event silo: the product should help Hosts, Venues, operators, connectors, and recurring rituals get discovered for funding, partnerships, sponsorships, grants, media, and custom workflows.

The default network posture should be:

```text
Being on Analog makes public-safe opportunity data discoverable,
unless the user, record, campaign, or compliance context opts out.
```

| Data class | Default posture | Guardrail |
|---|---|---|
| Public opportunity data | Discoverable through Analog search, Sponsor workflows, partner APIs, and Agent MCP. | Keep fields sponsor-safe: name, category, geography, cadence, audience hypothesis, funding needs, proof/readiness, offer preferences, and claim status. |
| Partner-supplied community data | Discoverable and usable when a partner platform has authority to list or route its downstream communities. | Preserve source, partner, consent, canonical-owner, and routing fields so Analog can send opportunity back to the app instead of pretending it owns the whole relationship. |
| Private operating data | Scoped to the relevant workspace, collab, partner, or campaign. | Do not expose partner-private notes, private messages, attachments, or sensitive strategy by default. |
| Attendee identity | Permissioned and purpose-limited. | Do not expose raw attendee emails, phone numbers, political affiliation, volunteer lists, donor data, or customer files as public opportunity data. |
| Sensitive funding, tax, or financial data | Restricted and approval-gated. | Use explicit authorization, purpose disclosure, and approved payment/tax/compliance workflows. |
| Public/nonprofit or political records | Visibility depends on program rules and legal/compliance posture. | Keep public reporting, political privacy, sensitive affiliation, and safety controls explicit. |

API, attribution, pixel, and Agent MCP access should make the opportunity graph usable from custom workflows: a Sponsor plan inside Analog, a partner platform dashboard, an external agent asked to source Sponsors for a Host, an event-page app reporting proof, or a city/operator workflow that routes a fund to matching supply. Agents and partner systems can read/query, report approved events, and prepare recommendations; they cannot publish, share private data, move money, send external outreach, or make final approval claims without human approval.

This is where a later platform-as-a-service motion belongs. Analog can sit behind event-page, ticketing, club, venue, nightlife, movement, creator, and community platforms as the Sponsor funding, proof, attribution, and opportunity-routing layer. Potential platform users include Partiful, Posh, Luma, Flite, LineLeap, MobiusFit, Eventbrite, and similar products that may prefer to keep their own attendee, organizer, venue, nightlife, fitness, or community experience while embedding Analog's sponsorship workflows.

Treat this as future product direction until partner API access, attribution ingestion, consent flows, billing, partner dashboards, approval controls, and support operations exist in production. Do not imply any named platform is currently integrated, committed, or a customer.

The user-facing bargain should be visible during onboarding, claiming, application, and profile settings: Analog is trying to expose qualified supply to relevant opportunities. Users should have opt-out, field-level visibility, private campaign modes, and correction paths where needed.

## Generated Supply Primitive

Generated supply is not fake inventory. It is new or newly fundable supply created when Analog removes a real blocker: money, proof, space, services, insurance, operator capacity, media visibility, or Sponsor fit.

Use [Market Formation](/strategy/market-formation) for the strategy doctrine. In product language, generated supply should still be represented through existing objects:

```text
Sponsor/public/community demand
-> RFE, RFC, fund, Event set, or Asset Need Intelligence
-> Host, Venue, operator, or partner response
-> collab, Host event, files/docs, spend, proof
-> catalog record and next allocation logic
```

| Generated supply shape | Current approximation | Rule |
|---|---|---|
| Latent Host activation | Campaign application, Host opportunity, pitch doc, city lead note, starter grant request | Use when a trusted person or group can start or restart a recurring ritual with small funding or support. |
| Missing-supply creation | RFE/RFC, Event set, Asset Need Intelligence, prospecting run, operator brief | Use when Sponsor or public demand exists but the right Host, space, service, proof, or operator is missing. |
| Venue-as-supply node | Host event location, Venue note, Venue reward, files/docs, proof packet | Treat Venues as fundable supply when space, mention coverage, local trust, or programming capacity matters. |
| Operator commission | Managed activation support, producer bench task, files/docs, campaign required todos | Keep bounded to a campaign, city, or fund; route learning back into templates and product objects. |
| Expanded proven supply | Recurring Host event, macro grant, operator retainer, cross-community collab, new Event set | Fund expansion only when prior proof, spend stewardship, Sponsor fit, or public value is clear. |
| Existing-supply pull | Claim flow, Host profile, regional dashboard, grant shelf, media inclusion, application portal | When generated supply gets funded through Analog, invite existing Hosts to claim records and compete for the next allocation. |

The product rule is simple: every generated supply motion must create product memory. If a funded experiment does not create a catalog record, Host profile, Asset Need Intelligence note, proof packet, no-go rule, or reusable playbook, it is drifting into bespoke agency work.

## Marketplace Matching Primitive

The marketplace primitive is not a public directory. It is the AI-assisted, human-reviewed logic that pairs Sponsor dollars with receptive Hosts, communities, Venues, event types, and proof standards.

Use [Marketplace Logic](/platform/marketplace) for the canonical Platform page.

```text
Sponsor category + goal + budget
-> campaign visibility mode
-> receptive Host/community/Venue/event-type match
-> governed spend
-> benchmarked proof
-> next-dollar decision
```

| Primitive | Current approximation | Rule |
|---|---|---|
| Campaign visibility mode | Campaign copy, application portal setting, outreach plan, event set prospecting, media launch | Every campaign should know whether it is private link-only, broadly marketed, or ICP-targeted. |
| Social receptiveness | Host notes, prior proof, content/mention history, Venue context, category fit, community exclusions, application answers | Match on whether the Sponsor is likely to be welcomed, not only whether the audience exists. |
| Category benchmark | Sponsor perspective, event type, proof packet, spend and mention history, assisted summaries | Compare success to organizations of the same category before recommending renewal. |
| Two-way review signal | Sponsor closeout reviews, Host closeout reviews, 90-day checkpoint notes, operator review notes | Use reviewed partnership quality to explain whether a partner is dependable, respectful, clear, and worth repeating. |
| Match reason | Deep research doc, Sponsor brief, event set note, catalog record, assisted summary | Explain why this Host, Venue, community, market, or event type fits the Sponsor's goal. |

This is future product direction until explicit visibility modes, receptiveness labels, review objects, automated benchmark libraries, and recommendation reason codes exist in production. Active `cc-analog` contributes directory, search, listing, claim, review, bookmark, volunteer, and grant-readiness signals. Preserved full-platform work can approximate the full loop with campaigns, event sets, application portals, prospecting, catalog records, files/docs, proof packets, closeout notes, and Sponsor data summaries.

## Two-Way Review Primitive

Two-way reviews are the structured human layer on top of AI proof and social listening.

Use [Two-Way Reviews](/platform/two-way-reviews) for the Platform rule. The default cadence is:

```text
every 90 days for active long-running collabs
or at closeout when funds are spent, the campaign ends, or the collab closes
```

| Primitive | Current approximation | Rule |
|---|---|---|
| Sponsor review of Host | Renewal note, proof QA note, Sponsor data summary, operator closeout file | Ask whether the Host was responsive, used funds as intended, delivered the activation, supplied useful proof, protected Sponsor fit, and deserves more budget. |
| Host review of Sponsor | Host note, task feedback, support ticket, operator closeout file | Ask whether the Sponsor was clear, responsive, respectful of the room, reliable with assets or funding, and worth working with again. |
| Funding history | Spend records, collab allocations, proof packets, closeout notes, verified external records where allowed | Show checks written by Sponsors and checks received by Hosts as counts, ranges, badges, or exact amounts only when permitted. |
| 90-day checkpoint | Task, reminder, file/doc, manual operator follow-up | Use for recurring collabs so issues surface before a long program quietly drifts. |
| Closeout review | Renewal packet, no-go note, proof packet, collab status note | Trigger when funds are spent, the campaign ends, or the collab closes. |
| Review visibility | File/doc permissions, internal note, Sponsor summary, Host summary | Separate private comments, reviewed-party feedback, internal Analog notes, future approval context, and public-safe aggregate summaries. |

The product should not become a public star board. It should turn review answers into review summaries, badges, reason codes, and confidence notes that support future Sponsor approval and Host opportunity decisions.

The trust loop is:

```text
proof + mutual review
-> reinforce good actors
-> constrain or remove bad actors
-> make future matching safer
```

Future product work should add review triggers, guided review screens, dimension ratings, visibility controls, summary generation, dispute handling, review-history cards, and matching reason codes.

## Cash Funding Ladder Primitive

Strategy pages may refer to a funding ladder. In the MVP, these are campaign/event-set conventions, not separate financial products.

| Funding ladder | Current product approximation | Rule |
|---|---|---|
| Micro Grant | Small collab allocation, spend card, event-scoped budget, or campaign application request | Use for USD 100 to USD 500/month style support: coffee, libations, supplies, rentals, print, accessibility, or light venue help. |

## Campaign Timing And Sponsor PLG Primitive

Campaigns should help Sponsors sense demand before they are forced into payment or heavy operations.

The product direction:

```text
future-dated campaign
-> public or targeted application response
-> first 48-hour supply signal
-> first five applicants visible on free
-> remaining applicants blurred or limited
-> first accepted applicant triggers paid plan, credit, campaign, or program flow
```

The start date is not only logistics. It is expectation-setting for Hosts and a demand-sensing tool for Sponsors. A Host should know whether the campaign is taking applications now for a future review or funding window. A Sponsor should be able to post a future-dated campaign, see whether the market responds within 48 hours, then decide whether to accept, narrow, broaden, or hold.

| Primitive | Current approximation | Rule |
|---|---|---|
| Campaign start date | Campaign copy, optional end date, application portal text, files/docs note, future campaign setting | Show applicants when review, funding, outreach, or activation is expected to begin. Do not let future campaigns feel like ignored applications. |
| 48-hour demand signal | Application count, source, fit notes, city/channel response, media or dashboard traffic | Use the first 48 hours after posting to judge whether the fund or campaign has pull before scaling operator work. |
| Free applicant preview | Sponsor application inbox, first five application cards, limited/blurred cards after five | Let Sponsors see enough supply to understand value without giving away the full market graph on free. |
| First accepted applicant gate | Campaign application acceptance, collab creation, budget setup, proof workflow | Do not paywall before value is visible. The conversion moment is first accepted applicant, before full collab operations and full applicant unlock. Route into a Sponsor plan with credits, a campaign commitment, or a program fee based on complexity. |
| Blurred applicant state | Limited card, aggregate count, high-level fit tags, source/channel summary | Do not expose sensitive Host details, full pitch language, contact info, attachments, or private social data before the Sponsor has committed. |

This is product direction until explicit start-date fields, free-tier applicant limits, blurred-card states, billing, and entitlement controls exist in production. Preserved full-platform docs can approximate the behavior through campaign copy, application portal language, application inbox handling, and operator rules.

Brand funds and programs are the cleanest market-making primitive: they give sponsors a reason to declare demand, hosts a reason to apply or create supply, and Analog a controlled environment to observe pricing, proof, spend behavior, and renewal.

Selective brand funds are the pre-marketplace version of this primitive. Analog should design a few Sponsor-funded public funds around specific markets before the open marketplace is liquid. The Sponsor supplies activation capital. Analog supplies fund-market fit, local distribution, application flow, regional dashboard placement, spend governance, and proof.

Two-way reviews are part of the same proof system. Future product specs should ask Sponsors and Hosts for structured reviews every 90 days in long-running collabs and at closeout when funds are spent, the campaign ends, or the collab closes. Review data should support future approval and matching, but it should use visibility controls, correction paths, and summary language instead of exposing every raw score or private comment.

- Host app: TanStack Router routes under `apps/analog-host/src/routes`.
- API: Fastify routes under `apps/analog-api/src/routes`, registered by domain at `/directory`, `/listings`, `/organizations`, `/storage`, `/notifications`, `/users`, `/blog`, and `/instagram`.
- Store: Mongo-backed listing repository and models in `packages/database`, with Local Atlas vector search for listing embeddings.
- Services: `packages/services` owns public DTO shaping, directory search, organization-owned listing workflows, grant applications, volunteer workflows, notifications, media/storage, and repository initialization.
- Active public DTOs must exclude owner organization ids, claim internals, Instagram token state, embeddings, internal notes, and future permissions/billing fields.
- Directory search stores lightweight `SearchConversation` records and uses bounded intent summaries, lexical search, and vector search; do not treat it as long-term AI memory.
- WorkOS AuthKit powers the active Host auth flow when configured.
- Shared GA4 tracking should keep event params low-cardinality and avoid emails, names, raw ids, tokens, or free-form form content.
- Archived total-platform code/docs preserve Sponsor campaigns, collabs, spend cards, Stripe Issuing, webhooks, Sponsor AI, files/docs, DMs, Gmail-shaped notifications, and full dashboard routes. They are not active production truth until restored.
- The MVP remains scaffolded for future auth depth, billing, payments, partner API, production attribution workflows, scoped AI memory, and full marketplace access.
