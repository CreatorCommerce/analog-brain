## Page Types

| Page type | Job | What belongs there |
|---|---|---|
| Overview | Orient the reader | Concepts, maps, decision framing, and links |
| Strategy | Explain doctrine | Market thesis, persona logic, sponsor fit, and operating posture |
| Reference | Own stable facts | Product objects, terminology, current MVP behavior, and invariants |
| How-to | Explain execution | Step-by-step workflows, operator checklists, and setup sequences |
| Use-case page | Route a test | What the use case tests and which canonical pages to use |
| AI playbook | Guide assistants | Update rules, prompt patterns, and conflict handling |

## Source Order

2. Current product behavior from the Analog MVP.
3. Analog strategy and market doctrine.
4. Existing summaries in this docs repo.

## Update Rules

- Update canonical references before summaries.
- Keep practical details in how-to pages.
- Link instead of copying long factual blocks across pages.
- Avoid bloat. Use visual components only when they reduce reading effort, and cut any card, tab, callout, table, or paragraph that repeats a decision already made clearer elsewhere.
- Keep stakeholder copy concrete. Employees, investors, early customers, hosts, city leads, and partners should see what Analog does, what proof matters, and what decision the page helps them make.
- Check light and dark mode when changing colors. Use true Analog red (`#FF4438`) as the light-mode accent. Do not use yellow as body text or primary navigation text on white backgrounds; reserve it for dark mode, fills, borders, or subtle accents. If red appears as small text on white, prefer the deeper readable red (`#B9362E`) or add enough surface contrast.
- Keep investor-facing language disciplined. Avoid guarantees, unsupported forecasts, raw PII exposure, unrestricted money language, near-term treasury/interchange claims, or wording that makes Analog sound like a custom events agency instead of a productized market-growth operating layer.
- Keep pricing language experimental until the founder makes a price official. Separate controlled activation budget from Analog revenue, and do not imply pass-through funds, unused balances, or future card economics are core revenue.
- When a new founder use case conflicts with existing language, revise the old language in the same pass.
- Add a release note when a new use case, workflow, rubric, or product rule is added.
