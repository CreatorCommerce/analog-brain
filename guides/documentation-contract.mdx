## Page Types

| Page type | Job | What belongs there |
|---|---|---|
| Overview | Orient the reader | Concepts, maps, decision framing, and links |
| Strategy | Explain doctrine | Market thesis, persona logic, sponsor fit, and operating posture |
| Reference | Own stable facts | Product objects, terminology, current MVP behavior, and invariants |
| How-to | Explain execution | Step-by-step workflows, operator checklists, and setup sequences |
| Use-case page | Route a test | What the use case tests and which canonical pages to use |
| AI playbook | Guide assistants | Update rules, prompt patterns, and conflict handling |

## Navigation Model

Use the tabs as a decision system:

| Tab | Job | What belongs there |
|---|---|---|
| Founding Docs | Orient the team and preserve source order. | Overview, stakeholder narrative, glossary, and AI maintenance docs. Source maps and release notes can stay linked but do not need top-level nav. |
| Market Education | Explain why the market exists. | Macro trends, tech-company landscape, social proof, external signals, and demand education. Do not put Analog's entry plan here. |
| Strategy | Explain how Analog enters the market. | Core thesis, why Analog wins, 12-month focus, market formation, supply-side MAC, cash funding, Sponsor/Host strategy, public/nonprofit and high-ticket lanes, pricing, GTM model, burn, media, and use cases. |
| Platform | Explain product truth and proof systems. | Current MVP objects, routes, proof, mention assets, Event Prospecting Catalog, Sponsor/Host workflows, spend, and product boundaries. |
| How-To | Explain execution. | Core step-by-step operator workflows. Specialty plays can stay linked from use cases instead of cluttering nav. |

Live navigation is curated. A page can remain published as a support page without appearing in top-level nav when it is narrow, referenced from a canonical page, or mainly useful to AI agents and operators.

## Length And Focus

Most live pages should be quick decision pages, not notebooks.

| Page kind | Target posture |
|---|---|
| Overview, Strategy, Use-case, How-to | Usually 60-125 lines. Lead with the decision, keep tables compact, and link to canonical pages instead of duplicating detail. |
| Reference and research pages | May be longer when they own stable product truth, source context, or evidence. Use headings and tables so readers can skim. |
| Image and evidence pages | May be longer when screenshots or source material are the point. Put interpretation after evidence when possible. |

If a page grows because it is trying to be strategy, reference, and how-to at once, split it or cut it.

Every live page must include:

```md
---
title: "Page title"
description: "One sentence on what this page covers."
pageType: "Reference"
---
```

## Update Rules

The core rule: founder-provided use cases, feedback, and corrections are law of the land.

- Update canonical references before summaries.
- Keep practical details in how-to pages.
- Link instead of copying long factual blocks across pages.
- Avoid bloat. Use visual components only when they reduce reading effort, and cut any card, tab, callout, table, or paragraph that repeats a decision already made clearer elsewhere.
- Write like an operator note: short sentences, concrete nouns, no filler, no vague "ecosystem" language unless the page names the actors and the next action.
- Keep stakeholder copy concrete. Employees, investors, early customers, hosts, city leads, and partners should see what Analog does, what proof matters, and what decision the page helps them make.
- Check light and dark mode when changing colors. Use true Analog red (`#FF4438`) as the light-mode accent. Do not use yellow as body text or primary navigation text on white backgrounds; reserve it for dark mode, fills, borders, or subtle accents. If red appears as small text on white, prefer the deeper readable red (`#B9362E`) or add enough surface contrast.
- Keep investor-facing language disciplined. Avoid guarantees, unsupported forecasts, raw PII exposure, unrestricted money language, near-term treasury/interchange claims, or wording that makes Analog sound like a custom events agency instead of a productized market-growth operating layer.
- Keep pricing language experimental until the founder makes a price official. Separate controlled activation budget from Analog revenue, and do not imply pass-through funds, unused balances, or future card economics are core revenue.
- When a new founder use case conflicts with existing language, revise the old language in the same pass.
- Add a release note when a new use case, workflow, rubric, or product rule is added.

## Update Workflow

1. Identify the canonical page that owns the fact.
2. Update that page first.
3. Search for conflicting language.
4. Update summaries, how-tos, scorecards, and glossary entries that now conflict.
5. Add a release note when the change creates a new use case, workflow, rubric, product rule, or strategic posture.

Before adding a page, ask how it strengthens the operating spine:

```text
market demand -> Sponsor capital -> Host supply -> governed spend -> proof -> next allocation -> product memory
```

If the page does not strengthen one of those links and is not needed for AI editing operations, merge it into a canonical page or keep it out of live docs.
