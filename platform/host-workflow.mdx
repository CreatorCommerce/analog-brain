Use this workflow when a Host has been accepted into a Sponsor campaign, has an active collab, or is creating event proof before sponsorship lands.

Read this after [Sponsor Workflow](/platform/sponsor-workflow) if you are learning the full exchange. Sponsor Workflow explains where the budget starts. This page explains how the Host turns it into proof.

The Host-side goal is not reimbursement with extra steps. The goal is approved funds, clear scope, no fronting money when possible, brand fit, a visible track record, and operator upside for dependable programs.

For the MVP, Host behavior is also marketplace signal. Clean spend, clear proof, and repeatable execution teach Analog which Hosts deserve more budget, which gatherings create value for attendees, and which support patterns should become product.

Hosts should ask for the support that actually makes the gathering better. That may be cash for coffee, libations, supplies, rentals, affordable space, print, or operator time; it may also be services/tools such as insurance, local design, vendor routing, contracts, or proof templates.

If sponsorship has not landed yet, the Host should still package the asset. A pitch page or Analog Doc should make event history, photos, social proof, community reach, prior Sponsor context, and the funding ask easy to understand.

The Host should also ask for non-cash support when that is the real unlock: photographer, videographer, print package, QR asset pack, insurance help, grant office hours, venue support, designer, or proof kit.

When the Host belongs to a priority market, the dashboard should feel local. A New York Host should see `AnalogNYC` and the funds, perks, grant paths, media requests, and proof tasks active in that market.

## 1. Build Host Trust Surface

Before or during sponsorship, keep the Host profile useful:

- host story
- socials
- Instagram connection when proof depends on social
- design kit
- files or pitch docs
- prior event history
- first-party data sources such as RSVPs, check-ins, claims, or opt-ins where permissioned

The Host pitch page and shared docs become proof that the Host can be trusted before a Sponsor relationship exists.

## 2. Check Sponsor Budgets

The Host should start from the Sponsors view or Events view and confirm:

- active city layer or market flag, such as `AnalogNYC` or `AnalogSF`
- local funds, public grants, raffles, or selective brand funds they can apply for
- available MAC perks such as photo/video, print, insurance, grant help, QR assets, or venue support
- active Sponsor relationships
- available budget
- allowed categories
- expiration
- whether the Sponsor card already exists
- event set assets or Sponsor files
- available services/tools support, such as print, docs, vendors, design, insurance, or proof kits
- whether the Sponsor added a Venue reward for mention coverage
- whether the campaign needs vendor/payee records, tax-document status, or 1099 support context for paid operators, Venues, contractors, or service providers
- special requests

## 3. Create The Host Event

Create a host event with:

- event name
- start and end time
- location or activation context
- Venue contact or authorization path when Venue mention coverage matters
- the Sponsor/collab funding the event
- approved spend categories
- proof requirements

The current event wizard can also create the event with no sponsor selected. Use that when the host is building credibility before a sponsor is attached.

## 4. Configure The Event Page

Use the event page when it has a real job: convert attendance, capture permissioned proof, or turn the gathering into a durable community asset.

Basic event pages are now commodities. A title, date, image, and RSVP button can help logistics, but it does not create much defensible value. Analog should care about event pages when they become a branded surface for the Host, the community, the Venue, and the Sponsor's role in making the gathering better.

The better standard is closer to a small Coachella or Outside Lands-style web experience for the community: high-context, visually owned, easy to share, and clear enough that someone wants to attend.

| Page mode | Use when | Standard |
|---|---|---|
| Basic capture page | The Host needs a quick RSVP, ticket model, QR destination, or proof link. | Keep it fast: status, theme, RSVP fields, ticket types if useful, sponsor-safe copy, and consent language. |
| Community brand asset | The gathering has Sponsor budget, media value, strong Host identity, Venue importance, or future repeat potential. | Use the Host design kit, stronger media, custom CSS/HTML where useful, sponsor role, Venue context, benefit/claim flow, QR assets, and proof prompts. |
| Future ad surface | The Sponsor or Host may later run paid distribution against the event, series, fund, or recap. | Capture clean links, UTMs, consent, RSVP/claim events, and page context so future ad constructs are built on a real IRL asset instead of a generic invite. |

Configure:

- page status
- theme and brand/design kit
- Host/community narrative
- Venue context when the space matters
- Sponsor role and attendee value, written as additive support
- RSVP fields
- ticket types if useful for modeling
- custom CSS or HTML when the event deserves a higher-production surface
- sponsor-safe copy and consent language
- QR or claim assets when the event needs first-party data or future Sponsor proof
- recap or proof path when the page should become event history after the gathering

## 5. Use The Spend Card

The Sponsor/Host spend card is attached to the collab and appears on the host event. Hosts should use it for approved operational spend only.

Examples:

- materials
- venue deposits
- food or beverage where approved
- production
- photo/video
- attendee benefit costs

The card is not a wallet or payout account. It is controlled activation spend that helps the Host execute and helps Analog learn which spend permissions are practical in the field.

If the activation uses paid Venues, photographers, artists, contractors, printers, caterers, or other service providers, keep the vendor record clean. Analog should ask once, remember the permitted record, and reduce repeated Sponsor-side tax and vendor chasing where possible.

The Host should not become the campaign's tax operations department. When a campaign needs W-9 status, payee context, or 1099 support, that should route through Analog's approved workflow or partner rail rather than scattered email threads.

For deeper recurring partnerships, the future path is a Host treasury account: an Analog checking-style operating balance for trusted Hosts and community businesses. That would let the Host pay vendors, venues, artists, staff, supplies, and recurring costs from one central rail while Analog keeps sponsor-safe proof, anomaly review, and renewal reporting.

If Analog creates a grade from bank or treasury activity, the Host should treat that grade as sensitive financial data. The Host must authorize account analysis and sponsor sharing, understand how the grade may affect future funding, be able to see the grade or key reasons, and have a way to dispute errors.

That is not current MVP behavior. Until those rails exist, Hosts should treat sponsor funds as controlled activation spend, not a withdrawable balance.

## 6. Request Sponsor Assets When Needed

If the sponsor attached event set assets, the host can:

- download them
- request mailed product, printed materials, samples, supplies, merch, or signage
- track whether the asset request is requested or mailed

## 7. Invite The Venue Into Mention Coverage

If the Sponsor added a Venue reward, the Host should make the ask simple:

```text
The Sponsor funding this event can also pay you a flat Venue reward
if you authorize social listening for the campaign window.
```

Use this when the Venue is part of the value chain: restaurant, studio, gym, gallery, hotel, coworking room, retail shop, bar, recovery space, or public-space partner.

The Host should clarify:

- the reward amount
- the campaign window
- which public or connected social accounts are in scope
- what the Venue needs to authorize
- that the reward is for permissioned proof coverage, not fake posting

This gives the Venue a reason to participate, gives the Sponsor better attribution coverage, and gives the Host more complete credit for the room they created.

## 8. Keep Proof Tight

The host should preserve:

- receipts
- vendor notes
- vendor/payee record status when requested
- event timing
- attendee counts
- content links
- story or post mentions
- Venue mentions when authorized
- issue notes
- RSVPs, attendance, check-ins, or claim data when available
- task completion and sponsor-required todos

## 9. Connect Instagram When Useful

Use Host Settings to connect Instagram when sponsor proof depends on post mentions or story mentions.

The product stores username history so older mentions can still resolve if the handle changes.

## 10. Prepare For Renewal

A strong host unlocks more budget by showing:

- the event happened
- the budget was used responsibly
- the audience or category grew
- the community welcomed the sponsor
- the host can earn for the work without weakening the gathering
- the sponsor received believable proof
- the host can repeat with less support

After each event, answer four renewal questions: did it create consumer or community value, was Sponsor money used responsibly, what proof would the Sponsor trust, and what would make the next activation cheaper or easier?
