Analog is the operating layer for sponsors to fund, support, and measure the real-world communities, creators, spaces, and operators that make markets.

The core belief is that digital saturation and AI abundance make real-world presence structurally more valuable. Sponsors need a governed way to fund IRL without turning every activation into a custom services project. Hosts need a practical way to get funded, prove the work, and earn again.

The sharper investor/operator line is:

```text
Invest in the positive upside to AI: more IRL.
```

The category-growth line is:

```text
Communities make markets. Brands monetize those markets.
```

The seed-deck operating line is:

```text
IRL is vibrant. But the economics are fragile.
```

Analog exists because demand, organizer work, and sponsor appetite already exist on both sides. The missing layer is the business model and operating system that makes sponsor participation safe, measurable, and renewable.

## Product Spine

The doctrine can be reduced to this loop:

```text
Predictive bet -> connected proof surfaces -> sponsor funds campaign -> host executes -> spend is watched -> proof is scored -> next unlock is recommended
```

For the current MVP, the product spine is simpler:

```text
Sponsor -> Campaign -> Event set / Application portal -> Host pitch -> Collab -> Host event -> Spend card -> Proof -> Unlock
```

The working product has also added the operating surfaces around this spine: files/docs, outreach, event set prospecting, hosted event pages, tasks, DMs, asset requests, shipments, brand kits, and sponsor data summaries.

From the sponsor and host perspective, the product should feel like:

```text
Discover trusted communities -> collaborate through programmable spend -> measure proof and outcomes -> renew or unlock more supply
```

## What The MVP Is Testing

The MVP should prove that:

- sponsors will fund controlled IRL activations when the budget feels governed
- sponsors can treat IRL as a trackable line item with approved hosts, categories, receipts, and reason
- hosts will use sponsor budget when it is simple, event-scoped, and not reimbursement chaos
- hosts can move from scattered side hustles to funded vocation when scope, spend, fit, and upside are clear
- sponsor applications can turn messy pitches into comparable opportunities
- spend cards can make small activations operationally safer
- proof can support renewal and underwriting decisions even before perfect attribution exists
- city-level supply work can create more sponsor-ready host options
- event sets can turn sponsor strategy into concrete prospecting and host-fit lanes
- hosted pages, imports, claims, and social proof can feed a privacy-safe audience graph
- files, docs, DMs, tasks, and shipments reduce operator load enough for the motion to repeat
- funding can reduce host burnout, reactivate dormant clubs, raise event quality, and convert more first-timers into regulars

## What Analog Is Not

Analog is not only an event directory, invite tool, ticketing platform, creator bank, social network, or reimbursement app.

The product should own the layer around the invite:

- sponsor budget
- host qualification
- campaign structure
- brief and approval logic
- spend governance
- proof
- renewal and unlock logic

## Mental Models

| Side | Should feel like |
|---|---|
| Sponsor | Spend management plus a market-growth underwriting cockpit |
| Host | Airbnb Host for funded IRL supply |
| Attendee | A normal event, claim, ticket, RSVP, or location benefit |

## Operating Rule

Market growth comes first. Direct attribution is valuable, but the stronger early signal is whether the activation grows category participation, content, recurring rituals, sponsor-ready supply, and host reliability.

## First Vertical Rule

Start where community already creates demand. Running and walking communities are the cleanest first vertical because they combine recurring weekly rituals, obvious uses of funds, dense supply, clear sponsor demand, and social proof that sponsors already understand.

Analog should not position around generic workflows too early. The first repeatable motions should become vertical-by-vertical practices, processes, and proof loops.

## Two Compounding Motions

Analog should keep two motions alive:

| Motion | Job | Product objects |
|---|---|---|
| Selective sponsor-funded pilots | Prove brand budget, spend governance, market-growth proof, and renewal | Campaign, event set, collab, host event, spend card, proof, data summary |
| Market formation | Create sponsor-ready supply before the market is liquid | Grants, RFEs, RFCs, hosted forms, catalog records, city leads, owned activations |

Both motions feed the [Event Prospecting Catalog](/market/event-prospecting-catalog) and [Asset Need Intelligence](/market/asset-need-intelligence).
